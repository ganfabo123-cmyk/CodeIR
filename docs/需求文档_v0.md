# CodeIR 需求文档 v0（供部署实现）

> 交付顺序 Step 1：需求文档——说清**做什么、为什么、在什么约束下做**。
> 本文档**不含实现代码**，作为 codex 部署的依据。配套《实验设计_v0.md》《数据格式参考_v0.md》。

---

## 1. 项目目标（为什么做）

验证一个方法论：**「专家模型协作 + 中间表示(CodeIR)压住级联信息损失」能否让小模型够到/局部超过大几倍模型的代码任务能力。**

- 试验台：LeetCode（仅 Python）。验证成立后方法可迁移到新任务（换 IR、换数据）。
- 核心要回答两个问题：
  - **问题A（机制，承重墙）**：IR 拆分架构本身有没有用？→ 实验臂 vs 同参数单模型对照。
  - **问题B（卖点，headline）**：系统整体能否够到大模型？→ 实验臂 vs 32B 参照线。

## 2. 范围与非目标

**在范围内**：
- 老师本地部署 + 蒸馏数据生成（含自适应 Best-of-N）
- 验证过滤（执行代码比对测试用例）
- 三条对照线的训练样本派生（实验臂 A、实验臂 B、baseline）
- LoRA 微调（LlamaFactory）
- 评测（pass@1，分难度；三条线 + 参照线；含 8bit 推理效率彩蛋）

**非目标（本期不做）**：
- 调试专家（模型C）——留接口，后续扩展
- 物理 4B+8B 非对称部署——future work
- 通用编程/非 LeetCode 任务

## 3. 硬件与环境约束（部署必须遵守）

| 项 | 约束 |
|---|---|
| GPU | **单卡 A6000，48G 显存** |
| 架构限制 | **Ampere，不支持 fp8** → 老师量化只能用 **int8**（bitsandbytes / GPTQ-8bit），**禁止 fp8** |
| 磁盘 | 现 51G，正在向管理员申请扩到 ~100G；部署需做磁盘预算 |
| 训练框架 | **LlamaFactory** |
| 老师模型 | **Qwen2.5-32B-Instruct，本地 int8**（≈34G 权重，48G 显存推理可容纳） |
| 学生基座 | **Qwen2.5-7B（≈8B），fp16**（不量化，≈16G，48G 显存 LoRA 训练轻松） |
| 老师调用 | 全本地，**¥0 API** |

磁盘预算（师生常驻）：老师 int8 ~34G + 学生 fp16 ~16G + 环境 ~12G + 数据/检查点 ~10G + buffer ~15G ≈ **~85G**（故申请 100G）。
若磁盘不足：老师阶段与学生阶段**分时跑、跑完删老师权重**，峰值可压到 ~50G。

## 4. 模块清单与职责（功能需求，codex 据此实现）

> 分层遵循全局规范：纯逻辑层（验证、派生）↔ 通信层（老师推理）分离。

### M1 老师部署与蒸馏
- **输入**：题库（题面 + 测试用例）。
- **行为**：本地起 Qwen2.5-32B int8 推理；对每题用统一 system prompt 生成 **Rich CodeIR + 代码**（格式见《数据格式参考》§1.R、§3）。
- **自适应 Best-of-N**：先贪心 1 次(temp=0)→ 送 M2 验证；AC 即收（sample_idx=0），WA 则 temp≈0.8 重采样至多 N=8，取首个 AC；全 WA 丢弃该题。
- **输出**：每题原始生成（含未过滤的，存 `teacher_raw/` 便于复盘）。

### M2 验证过滤执行器（纯逻辑，可独立测试）
- **输入**：(code, 该题 test_cases)。
- **行为**：真机执行 code，对每条用例比对返回值；全过且不超时 = AC。
- **安全**：子进程隔离 + `timeout_sec` 墙钟上限，防死循环/恶意代码。
- **输出**：`verified` 布尔；AC 的进 `verified_triples/`。

### M3 IR 排版/派生（纯逻辑）
- **目的**：解题贵、排版便宜 → **解耦**，IR 格式迭代不必重跑老师。
- **行为**：从 `verified_triples/` 的"正确解 + 推理"派生不同 IR 变体（Lean / Rich，见《数据格式参考》§1）。
- **输出**：各 IR 变体的结构化文本。

### M4 三条线样本派生（纯逻辑，同源切分）
- **铁律**：三条线样本**全部由同一份 verified 三元组派生**（杜绝提示词 confound）。
- 实验臂 LoRA_A：`(题面 → CodeIR)`
- 实验臂 LoRA_B：`(CodeIR → 代码)`
- baseline：`(题面 → CoT思路+代码)`，由 CodeIR **线性化**成行内 CoT（规则固定、可脚本化）
- **输出**：`sft_armA/ sft_armB/ sft_baseline/`（格式见《数据格式参考》§6）。

### M5 训练（LlamaFactory）
- 学生基座 **8B fp16**，LoRA 微调。
- **rank 容量对齐铁律**：实验臂 A=8、B=8；**baseline 单 LoRA=16**（=A+B）。
- 三条线**同基座、同精度、等训练算力**（GPU-小时计，非显存）。

### M6 评测
- **主指标**：pass@1（时间切分新题测试集，M2 执行判定）。
- **对照**：实验臂 / baseline / 32B参照线 三条线，**分 easy/medium/hard 报告**（不混总分）。
- **效率彩蛋**：额外报学生 **8bit（或4bit）推理** 的 pass@1（"压缩后照样赢"），两臂都报。
- **成本并列汇报**：实验臂推理跑两遍 8B（≈2×8B），须与精度提升一起列，禁止写成"8B 打赢 32B"。

## 5. 端到端数据流

```
题库(题面+测试)
   │  M1 老师 int8 生成 Rich CodeIR+代码（自适应 Best-of-N）
   ▼
teacher_raw/  ──M2 验证过滤(执行比对)──►  verified_triples/(全部AC)
   │
   ├─ M3 派生 IR 变体(Lean/Rich)
   └─ M4 同源切分
         ├─ sft_armA (题→IR)   ─┐
         ├─ sft_armB (IR→码)    ├─ M5 LlamaFactory LoRA(fp16, rank对齐)
         └─ sft_baseline(题→CoT+码)─┘
                                    │
                                    ▼  M6 评测(时间切分新题)
                          实验臂 / baseline / 32B参照线  × {easy,medium,hard}
                          + 8bit 推理效率彩蛋
```

## 6. 关键铁律（实验有效性，不可破）

1. **同源切分**：三条线样本来自同一份老师生成，只切分/线性化方式不同。
2. **容量对齐**：baseline rank = A+B（16）。
3. **同基座同精度**：三条线学生基座一致（8B fp16）；8bit 彩蛋两臂都做。
4. **防污染**：训练用老题；**测试集用时间切分新题，发布日须晚于老师知识截止日**。
5. **参照线诚实**：老师 int8 近乎无损；"打赢32B"按难度分档报，超不过老实报够到百分比。
6. **承重论点** = "≈2×8B 逼近32B + 拆分是主因"；超32B 仅当彩蛋。

## 7. 验收标准（怎么算成功）

- **问题A 成立**：实验臂 pass@1 > baseline（同基座/同数据/同信息/同容量下）→ 证明 IR 拆分本身有用。**这是项目成功的硬门槛。**
- **问题B 参考**：实验臂在 easy/medium 档逼近或超过 32B 参照线（hard 档够到多少如实报）。
- **每一步可独立验证**：M2 纯逻辑先用已知题自测；M1 老师调用先单题跑通；最小链路（单题 → 生成 → 过滤）先通，再规模化。

## 8. 里程碑（最小链路优先）

```
MS1：M2 验证过滤器单题自测通过（纯逻辑，不依赖老师）
MS2：M1 老师 int8 部署，单题生成 Rich CodeIR+代码
MS3：MS1+MS2 串通：单题 生成→过滤 一条完整路径
MS4：小批量(50题)产出 verified_triples，跑通 M3/M4 派生
MS5：M5 三条线各训一版(小数据)，跑通训练流程
MS6：M6 在小测试集跑三条对照线，看趋势
MS7：放大数据/调IR/开Best-of-N，逼近正式结论
```
