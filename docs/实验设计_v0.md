## 0. 一句话项目目标

验证一个方法论：**「专家模型协作 + 用中间表示(CodeIR)压住级联信息损失」能不能让小模型，够到甚至局部超过大几倍模型的任务能力。**

LeetCode（仅 Python）只是**试验台**，不是终点。验证成立 → 换 IR、换数据，方法可迁移到新任务。

---

## 1. 核心假设与思想

- **专家分工**：把"写代码"拆成认知阶段不同的专家，每个模型只干一件事，参数不浪费在无关能力上。
- **信息损失是天敌**：两段式 pipeline 的死穴是第一段丢失的信息第二段无法恢复（information bottleneck）。CodeIR 的全部意义，就是把"最难、最容易丢"的那部分（算法决策）**显式固定下来**，不让它在模型之间隐式流失。
- **可扩展**：核心是"专家协作"，不限于两个模型。后续可加调试专家等，本文档先锁定两段式。

### LeetCode 上的正确拆法（关键认知）

LeetCode 的题面已经结构化、无歧义，**没有"理解需求/应对人"的难点**。它的难点 100% 在**算法推理**（选 DP 还是双指针？状态转移是什么？边界怎么处理？）。

因此专家不按"人 vs 机器"拆，而按**解题的认知阶段**拆：

```
模型A（推理专家）：题面 → CodeIR（解题思路 / 算法选型 / 状态定义）   ← 真正难的一层
模型B（编码专家）：CodeIR → Python 代码                            ← 相对机械
[future] 模型C（调试专家）：代码 + 报错/WA用例 → 修正               ← 本期不做，留接口
```

### 现实预期（提前摆正，不赌输）

- **承重论点（稳）**：① 用 ≈2×8B 的有效算力逼近 30B 能力，成本低一截；② 消融证明强在"IR 拆分"本身，不是单纯蒸馏。
- **彩蛋（不押注）**：在 easy/medium 难度档上可能打平甚至小超 30B；hard 档大概率够不着（hard 靠深推理，8B 基座扛不住）。
- **铁律**：绝不把"全面超过 30B"当项目成败标准。超过了分难度报喜，超不过老实报"够到了百分之多少"。

## 2. 要回答的两个科学问题（别混）

| | 问题 | 跟谁比 | 性质 |
|---|---|---|---|
| **问题A** | IR 拆分这个**架构本身**有没有用？（机制） | 实验臂 vs 同参数单模型对照 | **承重墙，必须证** |
| **问题B** | 这套系统**整体能力**够不够得上大模型？（卖点） | 实验臂 vs 30B 参照线 | headline |

**关键区分**：
- 打赢 30B ≠ 证明是"拆分"的功劳——因为实验臂在任务上微调过、30B 没微调，赢可能纯是"微调过的小模型 > 通用大模型"（业界常识，不是贡献）。
- **必须靠问题A 的同参对照，才能把"赢"归因到拆分上。** 两个问题各管一件事，缺一不可。

## 3. 模型方案：8B 基座 + 双 LoRA

```
实验臂（IR方案）：8B基座(冻结) + LoRA_A(题→IR) + LoRA_B(IR→码)   ← 两个专家，切两段
对照臂（baseline）：8B基座(冻结) + LoRA_单(题→CoT+码)            ← 一个适配器包圆，不切
```

- 两个 LoRA = 两个专家行为，共享冻结基座（省显存），逻辑上就是"俩模型"，满足核心思想。
- **砍掉物理 4B+8B 非对称方案**：它只是省成本的产品包装，验证阶段引入"参数不对等"干扰变量。留作 future work（验证成功后再把 ModelA 缩成 4B 赚低延迟）。

## 4. 实验对照线（完整闭环）

```
对照臂B（baseline）：8B + 单LoRA，题 → CoT思路 + 码               ← 含推理、不拆、单模型
实验臂            ：8B + LoRA_A(题→CodeIR) + LoRA_B(CodeIR→码)    ← 结构化IR + 拆分
参照线           ：30B 老师自己在测试集上的得分                    ← 天花板
[可选] 强基线     ：给 baseline 单独配最优提示词的数据，赢了更硬     ← 鲁棒性补充
```

每条线证明什么：

- **实验臂 > baseline** → 在"同基座、同数据、同信息、同可训练参数"下，唯一差别是「结构化IR + 拆分」。赢了 = **架构本身有用**（问题A 结论）。
- **实验臂 ≈ / > 30B** → 系统整体够得上大模型（问题B headline）。
- baseline 设计为**含推理版（题→CoT+码）**，不用裸版（题→码）。否则赢了只能证"加了中间步骤有用"（可能只是 CoT 功劳），证不到拆分。

**注意**：A 的对照臂（8+单LoRA）**同时兼任** B 所需的"同参数单模型对照"。所以原计划的"单个 12B 模型"对照**不再单独需要**，结构因此简化。

## 5. 公平性控制（实验有效性的命根）

### 5.1 同源切分 —— 杜绝"提示词不公平"confound

**死穴**：若 IR 和 CoT 各用一个独立提示词，IR 提示词写得更细 → 数据更好 → 赢的是提示词不是 IR。

**釜底抽薪**：不追求"两个提示词一样细"（永远调不到真一样），而是**同一份老师生成，只切分方式不同**：

```
第一步：用一个提示词，大模型对每题只生成一次，产出完整 [推理 + 代码]
第二步：把这同一份生成，切成两种格式喂学生——
        实验臂 ：reasoning 整理成 CodeIR 结构 → 拆成 A(题→IR) + B(IR→码)
        baseline：同一份 reasoning 拉平成行内 CoT → 单模型 题→CoT+码
```

→ 老师一样、题目一样、推理内容一样、信息量一样，**唯一差别只剩"结构化+拆分"**。提示词 confound 物理消失。

### 5.2 可训练参数对齐 —— 堵"2个LoRA容量更大"confound

实验臂有两个适配器，baseline 一个，可训练参数 ≈ 2 倍。**修法**：

> baseline 单 LoRA 的 rank = 两个 IR LoRA 的 rank 之和（如 A、B 各 r=8，则 baseline r=16）。

总容量持平，唯一变量才真的只剩"拆不拆"。

### 5.3 等算力（iso-compute）

- **训练**：固定总训练算力，单位用 **GPU-小时（指明卡型）** 或 **总训练 token / FLOPs**，**不用显存当算力单位**（显存只决定跑不跑得起，不决定算了多少）。
- **推理（诚实汇报项）**：实验臂推理跑两遍 8B（先出IR再出码），实际推理量 ≈ 2×8B ≈ 16B。
  - 不影响"机制有没有用"的精度结论。
  - 但汇报时**必须把"精度提升"和"推理成本 ≈2×8B"一起列**，禁止写成"8B 打赢 30B"，应写"≈2×8B 推理逼近 30B"。好消息：16 < 30，"便宜"卖点依然成立。

### 5.4 防数据污染

- 30B 很可能背过老 LeetCode 题。**训练数据**用老题无妨；**测试集必须用时间切分的新题**（推荐 LiveCodeBench 按时间切），否则"反超 30B"可能只是两个都背过答案。

## 6. 数据流 / 数据构造

```
[题库(题面 + 测试用例)]
        │
        │  用 30B 蒸馏（同一个老师贯穿 A、B，杜绝"AB数据不同"刁难）
        ▼
[30B 对每题生成：CodeIR + 代码]   ←（可选）Best-of-N：每题采样N次
        │
        │  ★ 验证过滤（LeetCode 杀手锏）：拿题目测试用例真机跑代码
        ▼
  ┌─ AC（全过） → 留下，进训练集
  └─ WA/错误   → 丢弃
        │
        ▼
[干净三元组：(题面, CodeIR, 代码)，全部验证正确]
        │
        ├─ 实验臂：切成 (题→IR) 训 LoRA_A、(IR→码) 训 LoRA_B
        └─ baseline：拉平成 (题→CoT+码) 训单 LoRA
```

### 为什么必须验证过滤

纯模仿学习天花板 ≈ 老师本身。老师也会做错题，把对错样本全喂学生 → 学生学进错误 → 永远超不过老师。验证过滤后学生只学**老师做对的子集**，擦掉噪声 → pass@1 才有机会反超老师平均水平。**这是反超 30B 的唯一本钱，也是 LeetCode 白送的能力**（代码对错可执行验证，不靠人判断）。

## 7. CodeIR 设计（预设草稿，待按效果迭代）

定位：**算法思路的结构化表示**——把最难、最容易丢的"算法决策"显式固定。

```yaml
pattern: dynamic_programming          # 算法范式，枚举值（dp/two_pointers/bfs/greedy/binary_search/hashmap/...）
key_insight: 一句话点破本题最关键的算法观察   # 最重要字段，最该被固定
state: dp[i] 的定义 / 核心数据结构
transition: 状态转移 / 核心步骤
complexity: { time: O(n), space: O(n) }   # LeetCode 常硬性要求复杂度
edge_cases: [空输入, 单元素, ...]
```

示例（最长递增子序列）：

```yaml
pattern: dynamic_programming
key_insight: 以每个元素结尾的最长上升子序列长度可由前面更小元素的状态转移得到
state: dp[i] = 以 nums[i] 结尾的最长递增子序列长度
transition: dp[i] = max(dp[j]+1 for j<i if nums[j] < nums[i])
complexity: { time: O(n^2), space: O(n) }
edge_cases: [空数组返回0]
```

- **[待确认] IR 是否内嵌思维链**：已定为"纯 CodeIR，但允许把思维链揉进 IR 字段（如 key_insight/transition 写得更展开）"。具体展开到什么程度，按效果调。
- response schema 遵循全局规范（顶层结构约束，字段描述放 system prompt，避免深层嵌套）。

## 8. 评估指标

- **主指标**：pass@1（时间切分新题测试集上，代码真机跑测试的通过率）。
- **分难度报告**：easy / medium / hard 分开报，不混成一个总分（hard 大概率拉垮，混报会掩盖 easy/medium 的真实增益）。
- **成本指标（与精度并列汇报）**：推理 FLOPs / 单题延迟（实验臂记 ≈2×8B）、训练 GPU-小时。
- **对照呈现**：实验臂 / baseline / 30B 参照线三条线，每个难度档一行。

## 9. 最小验证链路（先跑通一条，再扩展）

> 遵循全局规范"先跑通最小链路，纯逻辑优先，外部依赖单独验证"。本节只描述路径，**不写实现**。

```
Step 1：选 1 道 medium 题，手写一份 CodeIR + 代码，本地真机跑测试用例 → 验证"验证过滤"这条管道能跑通
Step 2：调 30B API，对这 1 道题生成 [CodeIR + 代码] → 验证蒸馏提示词 + response schema 能出干净结构
Step 3：把 Step 2 接上 Step 1 的执行验证 → 跑通"生成→过滤"一条完整路径（单题）
Step 4：扩到小批量（如 50 题），产出第一批干净三元组 → 验证数据规模化没问题
Step 5：最小训练——LoRA_A、LoRA_B、baseline 单 LoRA 各训一版（小数据先跑通流程）
Step 6：在时间切分小测试集上跑三条对照线，看趋势是否成立
Step 7：放大数据 / 调 IR / 加 Best-of-N，逐步逼近正式结论
```

| 7 | 训练 / 硬件 | **单卡 A6000 48G + LlamaFactory**。学生 8B fp16 + LoRA（≈16G，48G 显存轻松）。老师 32B int8（≈34G，推理可容纳）。 |

### 硬约束备忘（A6000 48G 显存 / 磁盘申请 ~100G）

- **A6000=Ampere，不支持 fp8** → 老师量化只能 **int8**（bitsandbytes / GPTQ-8bit），禁止 fp8。
- 32B fp16 ≈ 64G 装不下 → 老师用 int8 ≈ 34G（显存/磁盘可容纳）。
- 学生 8B **fp16**（满血，最优微调质量）；**效率彩蛋**额外报 8bit/4bit 推理分数（"压缩后照样赢"）。
- 精度对称无科学意义：老师 int8 是被逼（32B装不下），学生 fp16 是因为装得下就该满血。两臂学生须**同精度**。
- 磁盘预算（师生常驻）≈ 85G → 申请 100G；不足则老师/学生分时跑、跑完删老师，峰值 ~50G。
